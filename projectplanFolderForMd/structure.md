# SOUSCHEF — PROJECT SKELETON (MONOLITH)

A proposed repository structure for the project defined in [amer_idea-1.md](amer_idea-1.md),
[amer_idea-1-tech.md](amer_idea-1-tech.md), and [model_role.md](model_role.md).

**It's a monolith.** One FastAPI application holds every layer — `api → services → repo → infra` — and
the classifier, guardrails, agent, and constraint guard live *inside* it as internal packages (called
in-process, not over the network). The React widget, the Streamlit dashboard, the offline ML training,
the data ingestion, and the evals share the same repo but are separate top-level folders. The only
things that run as their own containers in `docker-compose` / Railway are the **stateful services** the
app talks to (Postgres, Redis, Vault, Phoenix) — not pieces of the app itself.

**Principles reflected in the layout**
- *Layered, auditable backend* — all DB access in `repo/`, all external calls behind `infra/` adapters.
- *Grounding & safety in code* — the constraint guard and guardrails are their own packages with tests.
- *Train heavy, serve lean* — training lives in `ml/` (never shipped); serving loads a `.joblib` artifact.
- *Specs before code (SpecKit)* — `.specify/` + `specs/` hold the source-of-truth specs.
- *Evals are the grade* — `evals/` golden sets + `eval_thresholds.yaml` gate CI.

## NOTES ON A FEW DELIBERATE CHOICES

- **Why one `app/` package, not microservices.** The plan rules out service sprawl. The classifier,
  guardrails, agent, and constraint guard are *modules* of the monolith, called in-process — simpler to
  build, test, trace, and reason about for one engineer. The only separate containers are stateful
  backing services (Postgres, Redis, Vault, Phoenix) and the two *other surfaces* (Streamlit dashboard,
  React widget), which are genuinely different apps.

- **`services/` is split by audience: `user/` vs `admin/`.** Cook-facing logic (search, RAG, the wall,
  meal planning, favorites) lives in `services/user/`; operator-facing logic (corpus, evals, metrics,
  ingestion, traces) lives in `services/admin/`. The split is mirrored in `api/user/` and `api/admin/`,
  and `api/admin/*` is gated by `admin_deps.py` (operator auth) — so the public widget can never reach
  an admin service. Anything both sides need goes in `services/shared/` to avoid duplication. This is
  still one monolith (same process, same deploy); the folders express *who a service is for*, not a
  network boundary.

- **The Streamlit dashboard is the admin UI; `services/admin/` is its brain.** The dashboard calls the
  `api/admin/*` endpoints, which call `services/admin/*`. Keeping that logic in the backend (not in the
  Streamlit scripts) keeps it testable and reusable.

- **`repo/` is the only place that touches the DB.** That's what makes the freshness rule and the
  constraint queries auditable, and it keeps SQL parameterized (injection-safe) by construction.

- **`infra/` isolates everything external** (Groq, embeddings, Vault, Phoenix, the data APIs) behind
  adapters, so tests mock them and a provider swap is a one-file change.

- **`ml/` never ships.** Training (heavy) is separate from serving (lean); the container only loads the
  exported `model.joblib`. No torch anywhere in `app/` or the image.

- **Redaction sits in `core/` and is used twice** — by logging and by `infra/tracing.py` before any span
  reaches Phoenix — so a pasted secret can't leak through either path.

- **SpecKit drives the build**: `.specify/` + `specs/` are written first; the code under `app/` is
  generated/implemented against those specs and reviewed.
